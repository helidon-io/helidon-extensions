# Agentic Chess Design

## Status

Active. V1 example implementation is in progress, and the design is being kept aligned with the implemented behavior.

## Purpose

Define the scope, architecture, UX, dependencies, and validation strategy for a new LangChain4j example project at `langchain4j/examples/agentic-chess`.

## Project Summary

- Project name: `agentic-chess`
- Location: `langchain4j/examples/agentic-chess`
- Repository area: `langchain4j/examples`
- Current state: v1 product requirements captured and architecture proposed

## Goals

- Demonstrate Helidon LangChain4j agentic capabilities with a chess-focused example.
- Produce an example that is understandable, runnable, and useful as extension documentation.
- Keep the initial implementation aligned with repository conventions for examples.
- Show Helidon developers how to properly use LangChain4j in Helidon SE declarative applications.
- Prioritize a rich user experience with a purpose-built chess UI.

## Non-Goals

- No multiplayer in v1.
- No durable persistence in v1.
- No custom position loading in v1.
- No external chess engine, opening database, or MCP integration in v1.

## Target Audience

Helidon developers who want to learn proper LangChain4j usage patterns in Helidon.

## Primary Use Case

The user plays a full chess game against an AI opponent through a web UI.

## User Experience

- The example exposes a web UI with a chessboard and supporting UI components.
- Chess pieces animate when moves are made.
- When the human-in-the-loop step is waiting for user input, the board highlights legal user choices.
- The user selects a piece first and then chooses from legal destination squares.
- The user provides input by clicking on the chessboard rather than typing raw move notation.
- The page should automatically start a game on first load and resume the current browser session on refresh.
- The UI should show board state and candidate lines generated by the AI.
- The UI should include a latest-commentary panel and a candidate-lines panel beside the board.
- The commentary panel should show only the latest commentary item, with move coordinates visually highlighted.
- The UI should show a visible progress indicator while Black is thinking or commentary is still streaming.
- After a human move, White-move commentary may stream while Black is already evaluating its reply.
- The candidate-lines panel should be scrollable and show the newest candidate batch first.
- On desktop, the candidate-lines panel should remain visually contained within the same vertical pane as the main board card.
- The main board should use fixed-size square cells inside a rounded outer frame.
- The human player always plays white in v1.
- The custom chess UI lives under `/chess`.

## Functional Requirements

- Support a full chess game between a human and an AI opponent.
- Use a human-in-the-loop step to request and accept the human player's move.
- Preserve game state across turns for the full match.
- Use per-browser-session game state for v1.
- Automatically create or resume a browser-session game when the page loads.
- Render the current board position in the browser.
- Present user-selectable move options directly on the board.
- Generate and display the top 3 candidate lines for the AI side.
- Validate all legal moves deterministically in application code.
- Reject or reprompt illegal AI moves using LangChain4j guardrails.
- Start from the standard chess starting position only in v1.
- Support a live OpenAI-compatible provider configuration.
- Support mock-model mode for tests and offline validation.
- Run as a Helidon SE declarative example application.

## Agentic Design

### Workflow Shape

- The flow must include a human-in-the-loop interaction.
- The human player does not type free-form move text; the workflow pauses for explicit move selection in the UI.
- Preferred v1 direction is a turn-based workflow in which deterministic chess state management surrounds the agent calls.
- The human-in-the-loop step is used to model the player move collection as an explicit workflow stage, not as a raw REST parameter passed straight into the AI move generator.

### Agents

- Required:
  - AI opponent agent
  - human-in-the-loop move collection agent or equivalent workflow step
- Preferred v1 direction:
  - a game orchestration agent or service coordinates turn flow
  - an AI move agent proposes the opponent move and candidate lines
  - a separate commentary-producing agent explains the current position after both human and AI moves

### Tools

- No external tools are planned for v1.
- Keep a design note for possible future use of tools such as chess engines, databases, or MCP integrations in a later version.
- TODO for future version: evaluate whether a chess engine or MCP-exposed chess service should be added for stronger analysis and validation beyond the didactic v1 example.

### Model Strategy

- The live provider target for v1 is the OpenAI-compatible provider.
- Planned provider configuration:
  - `langchain4j.models.<model>.provider: open-ai`
  - `langchain4j.providers.open-ai.base-url: https://inference.generativeai.us-chicago-1.oci.oraclecloud.com/20231130/actions`
  - `langchain4j.providers.open-ai.api-key: ${OCI_API_KEY}`
- Default model name: `openai.gpt-oss-120b`
- Mock-model support is required for tests and offline validation.
- The AI opponent should use a typed LangChain4j structured-output DTO return value instead of hand-written JSON schema embedded in the user prompt.
- For the current OCI OpenAI-compatible live path, the example should rely on LangChain4j's automatic fallback output-format instructions for typed DTO parsing rather than enabling provider-side JSON-schema response formatting by default.

### State and Memory

- The example must maintain full-game state across turns.
- The state model needs to track at least:
  - board position
  - side to move
  - legal moves currently available to the human player
  - move history
  - candidate lines
  - commentary history
  - whether AI move selection is currently running
  - whether commentary streaming is currently active
  - the active commentary phase label for reconnect/resync
  - session or memory identifier
- Preferred v1 direction is an in-memory server-side session store keyed by a browser session identifier, with no durable persistence.
- LangChain4j memory may still be used for agent context, but authoritative chess legality and board state should live in deterministic application state rather than only in chat memory.
- Session state concurrency uses explicit `ReentrantReadWriteLock` coordination, and WebSocket outbound sends use per-session `ReentrantLock` serialization instead of intrinsic monitors.

## Chess Domain Scope

### Supported Capabilities

- Full human-versus-AI game flow.
- Board rendering in the browser.
- Candidate line display.
- Human move entry by clicking a suggested move on the board.
- Latest-commentary and candidate-line display.

### Rules and Validation

- Strategic reasoning is mostly LLM-driven.
- Legal move generation must be deterministic in application code.
- The deterministic rules layer must be authoritative for:
  - user move highlighting and selection
  - move application
  - illegal AI move rejection
- Full legal move support is required for v1.
- Guardrails should reprompt the model when it proposes an illegal or malformed move.
- Deterministic terminal-state handling in v1 is limited to checkmate and stalemate.
- Pawn promotion in v1 always promotes to queen.

### Game Representation

- It must support full-game progression and browser rendering.
- It should support deterministic rules processing and clean JSON exchange with the browser.
- Internal representation is still open, but should likely separate:
  - domain board state
  - user-facing board rendering data
  - AI-facing move/context representation
- Starting position only in v1.

## Proposed Architecture

### Preferred Module Shape

- `langchain4j/examples/agentic-chess/pom.xml`
- `src/main/java` for Helidon services, chess domain model, LangChain4j agents, and REST/UI endpoints
- Preferred package layout:
  - `...chess.ai` for agents, HITL/LLM support, memory, and guardrails
  - `...chess.rest` for declarative HTTP and WebSocket endpoints plus transport-side event broadcasting
  - `...chess.engine` for chess rules, session state, orchestration, and engine model types
  - `...chess.dto` for JSON/API and event payload types
- Keep the application entry point as `...chess.Main` in the root example package.
- `src/main/resources` for configuration and static chess UI assets

### Preferred Runtime Components

- Helidon static-content feature
  - serves the custom chess page and assets from the classpath under `/chess`
  - uses a welcome file so the browser lands on `/chess/`
- `ChessApiEndpoint`
  - exposes JSON endpoints for game lifecycle and move submission
- `ChessSessionStore`
  - in-memory game/session store for v1
  - keyed by a browser session or generated game identifier
- `ChessSession`
  - protects mutable game state with `ReentrantReadWriteLock`
  - exposes write-locked snapshot generation for coordinator transitions
- `ChessRulesService`
  - authoritative deterministic chess rules implementation
  - computes legal moves and applies accepted moves
- `ChessGameCoordinator`
  - coordinates turn boundaries between browser input, deterministic state, and LangChain4j agent calls
  - runs White commentary and Black move selection in parallel after an accepted human move
- `ChessOpponentAgent`
  - returns a typed move-proposal DTO with the AI move and top 3 candidate lines
- `ChessCommentaryAgent`
  - produces commentary for the accepted move and current position
- `ChessTurnWorkflowAgent`
  - models a full turn workflow that includes the human-in-the-loop move collection stage and the opponent move selection stage
- `ChessHumanMoveCollectorAgent`
  - provides the dedicated human-in-the-loop collection step used by the turn workflow
- `ChessHumanMoveTransitionAgent`, `ChessTurnOutcomeRouterAgent`, and `ChessTurnResultAssemblerAgent`
  - keep the workflow's deterministic transition, conditional opponent routing, and final turn-result assembly in standalone top-level `*Agent` types
- `IllegalMoveOutputGuardrail`
  - validates the AI proposal against the legal moves from `ChessRulesService`
  - reprompts on malformed or illegal output while normalizing the structured result
- `ChessEventBroadcaster`
  - serializes outbound WebSocket sends per browser session using explicit reentrant locks instead of intrinsic monitors

### Preferred Turn Flow

1. Browser opens the chess page at `/chess`.
2. Browser restores the current browser-session game if one exists; otherwise it starts a new session automatically.
3. For a new session, the server creates an in-memory game and computes legal white moves from the standard initial position.
4. Browser renders the board, the latest-commentary area, and the candidate-line area, with the candidate pane scrolling within the board-column height on desktop.
5. User clicks a white piece, then selects one of the legal destination squares offered by the UI.
6. Server validates and applies the human move through the deterministic rules layer.
7. After a legal human move, commentary for White's move starts streaming.
8. While that commentary is streaming, the server invokes the AI move flow for Black in parallel.
9. AI move output is checked by a guardrail against deterministic legal moves.
10. If the AI move is invalid, the model is reprompted.
11. Once White-move commentary has finished, the accepted legal AI move is applied and the top 3 candidate lines are stored.
12. Commentary for Black's move then streams against the updated board state.
13. Browser receives streamed events and final state updates, then animates the move.

### Preferred API Shape

- A lightweight JSON API is preferred over coupling the custom chess UI to the shared LangChain4j dev UI endpoints.
- Likely endpoints:
  - create game
  - fetch current session state
  - submit human move
  - reset game
- The runtime should support streaming updates for AI turns and human-in-the-loop state transitions.
- Preferred transport is a declarative WebSocket endpoint.
- The stream should carry all relevant game events, including:
  - human-in-the-loop waiting state
  - move validation outcomes
  - AI turn progress
  - concurrent AI-thinking and commentary-streaming progress state
  - accepted AI move
  - commentary update
  - final game-state update
- The design explicitly requires reconnect/resync behavior after browser refresh or transient WebSocket disconnect.
- On reconnect, the browser should be able to recover the authoritative current game state for the active browser session.
- The event broadcaster should tolerate stale or abruptly closed WebSocket sessions and evict them without breaking the game loop.

## Testing and Validation

- Must be suitable as a repository example and therefore should have a clear validation story.
- Mock-model mode is required.
- Deterministic chess rules should be testable independently of the model provider.
- Example validation should not require durable infrastructure beyond the configured model endpoint for the live path.
- V1 test scope is unit and server-side testing only; no browser automation is required.

## Documentation Expectations

- The example should teach correct Helidon plus LangChain4j usage, not just provide a demo.
- Documentation should explain the agentic flow, the human-in-the-loop step, and the UI interaction model.
- Documentation should explain why deterministic rules are still necessary even though the strategic reasoning is mostly LLM-driven.
